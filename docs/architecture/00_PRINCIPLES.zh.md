# TourSkill 架构原则

> *"做未来的产品，不贪图当下方便。"* —— 北极星，2026-04-29

这份文档是其他每一份架构文档都会回指向的"宪法"。当两份文档冲突时，
原则优先。当一个未来 PR 看起来"方便但不对劲"时，先对照这份清单
做自检。

---

## 原则 1 —— 标准优先，绝不 fork

我们**原样**采用已发布的规范，并将系统集成进去。

- **ERC-8004**：链上 agent 身份/信誉/验证三联 registry。
- **A2A Agent Card**：链下 JSON 描述符，把链上身份桥接到一个可调用的 agent endpoint。
- **x402**：HTTP 原生的支付握手协议（Coinbase 主导，Base 原生）。
- **EIP-191**：链下签名验证（我们的 auth 已经在用）。
- **OpenAI tool-call wire format**：agent 与 skill 之间的协议（所有主流 LLM 厂商都支持）。

我们**不**为了"改进"去 fork 这些规范。如果遇到真实限制，要么向上游
提议变更，要么使用规范自带的扩展机制（例如 agent-card.json 的
`extensions` 字段）。

**为什么重要**。一个**别人写的客户端也能消费**——比如 Coinbase 钱包、
erc8004.org 索引器、或者任何未来的 agent——这才让我们成为**协议**而不是
**平台**。一旦 fork，我们就变成另一个围墙花园。

---

## 原则 2 —— 身份与状态解耦

两层、两种存储、一座密码学桥。

```
┌────────────────────────────────────────────────┐
│  身份（不可变、低频）                          │
│  → 链上（Base Sepolia / mainnet）              │
│  → ERC-8004 三套 registry                      │
│  → 存：agent 地址、agent_card_uri、            │
│        profile_hash、注册时间戳                │
└────────────────────┬───────────────────────────┘
                     │  SHA-256(profile.json) commit 上链
                     ▼
┌────────────────────────────────────────────────┐
│  状态（可变、高频）                            │
│  → 链下（每个 merchant-agent 自己的数据库）    │
│  → agent-card.json + skills + 库存             │
│  → 任何每日变化的东西都在这里                  │
└────────────────────────────────────────────────┘
```

**我们运行的链下 Supabase 数据库是 cache + indexer，不是事实来源**。
如果我们的 Supabase 明天挂了，每个 merchant-agent 仍然拥有自己的
完整状态，链上的哈希仍然能证明他们的身份。任何人都能通过回放链上
事件重建我们的索引器。

**含义**：任何 TourSkill 后端功能如果**依赖**我们的中心化数据库才能
运行（不是仅仅为了性能），就是味道不对的信号。

---

## 原则 3 —— 没有 mock 执行

注册表上广告的每一个 skill 都必须能 trace 到一个真实的 merchant-agent
endpoint，那个 endpoint 拥有自己的数据。

我们当前的 `skill_service.py`（12 个伪造数据的 handler）已经被列入
砍除清单。Phase A 之后：

- TourSkill 后端**不执行**任何 merchant skill。
- 它只做**发现**（registry indexer）、**认证**（challenge-response）、
  **路由**（把 `agentURI` 翻译成一个 HTTP 调用）。这就是它工作的全部。
- 一次 skill 调用从 user-agent → 直接到 merchant-agent，**不**经过
  我们后端的业务逻辑。我们的后端在 platform-hosted 商家的网络路径上
  会出现，但它**转发字节，不合成字节**。

**为什么重要**。"中心化后端代替假商家执行 mock"恰好就是我们指控
OTA 干的事。一旦我们留下这条捷径，**我们就是他们的同类，只是套了
个新壳**。

---

## 原则 4 —— 净身重来 优先于 向后兼容

当现有形状是错的，我们删掉它。我们不会因为惯性而保留错误。

具体应用（2026-04-29）：

- 0G 测试网上 legacy `MerchantRegistry.sol` 注册的 28 个 seeded 商户
  是假的。**不会**迁移到 ERC-8004。我们在 Base Sepolia 上**从零**
  重新注册一组更小的、真实的参考 agent。
- legacy 合约地址保留在链上（链历史不可变），但我们的应用停止读它。
  在文档里标记为 deprecated。
- 那 28 家在 Supabase 里的行**保留**（在开发新 merchant-agent 模板
  期间作为确定性测试 fixture 有用），加 `legacy_seed: true` 标记。
  新系统上线后，**永远不会**被公开发现接口暴露。

**为什么重要**。"写迁移脚本把旧数据搬到新形状"听起来很负责，但
**比净身重来花更多工程量**——尤其是当旧数据本来就是错的。我们应该
**不背历史包袱地为未来设计**。

---

## 原则 5 —— 一个协议，两条链

身份与支付的访问模式不同。强行塞到一条链上是无意义的税。

| 层 | 链 | 原因 |
|----|------|------|
| 身份（ERC-8004）| **Base Sepolia → Base mainnet** | erc8004 索引器/钱包/工具集中在 Base；Coinbase 把这个标准推到了 Base；与 x402 同链 |
| 支付（x402 + escrow）| **Base Sepolia → Base mainnet** | 原生 USDC、成熟的 Smart Wallet、x402 SDK 是 TS 原生 |
| AI 推理计费 | **0G Network** | 保留现有 `use0gCompute`；0G 专为这个建造；与商家身份正交 |
| AI 存储（未来）| **0G Storage** | agent 训练数据、模型权重、大型 profile blob |

注：身份+支付当前都落在 Base 是一个 **当前生态系统时刻**的偶然——
不是"永远两条链"。原则是"对的工具放对的地方，当值得时接受运维成本"。

**含义**：TourSkill **不**会为了让"0G 项目"故事更整洁就把所有东西
推到 0G 上。我们对 0G 的实际用例是**推理 + 存储**，那才是 0G 真正
擅长的。身份和支付该去哪条链就去哪条链。

---

## 原则 6 —— 两轨托管，一个外部接口

商家有两种形态：

- **自托管**：fork `merchant-agent-template` 仓库，自己跑在自己的基础
  设施上，自己掌握私钥，自己付服务器钱。**最大主权，最高技术门槛**。
- **平台托管**：TourSkill 帮他们跑 agent，多租户，按 SaaS 订阅付费。
  **最低门槛，需要信任我们**。

**硬规则：从外部看，两种部署完全相同。**

- 两种都在同一规范路径上服务 agent-card.json。
- 两种都用自己的钱包注册到 ERC-8004（**我们从不托管私钥**）。
- 两种都暴露完全相同的 skill endpoint，使用完全相同的 wire format。
- 两种都用完全相同的 challenge-response 给 user-agent 认证。

这条规则让我们诚实：自托管的商家相对平台托管的，**永远不该感到二等
公民**。如果某个特性只在平台侧能用，它该回到上游 Hono / 中间件，
而不是塞在我们专有的 fork 里。

---

## 原则 7 —— 用户确认门，把守不可逆动作

任何 agent ——无论 user-side 还是 merchant-side ——在没有人类显式
确认之前，**都不能完成不可逆的动作**。

- **预订、支付、列表变更、profile 编辑**全部要求受影响的人类看到
  diff 并点"确认"才签。
- 对于代替不在场 owner 工作的 agent（"Sign Once, Govern Forever"
  模式），我们用短期 bearer token 限制爆炸半径（今天 30 天 TTL；
  可撤销；未来 SIWE 绑定）。
- **幻觉作为一等威胁模型对待**，不是 bug。

**含义**：任何 agent 可能"自主决定就做了"的地方（转账、签合同、
发表公开评价），架构都强制路径上有一个人类可读的确认步骤。

---

## 原则 8 —— 可持续 优先于 速度

我们不是为了下周的 demo 在建。我们是为了明年的生产在建。当一个捷径
会创造**复利型技术债**时，我们走更慢的路。

具体表征：
- 我们依赖的每一个外部 API（链 RPC、LLM provider、Supabase……）
  都是可通过 config 替换的；**没有 provider 被硬编码到业务逻辑里**。
- 每一次状态突变都能从链上事件 + agent-card 快照重新生成。
- 每一个特性都有"如果这个 provider 明天消失会怎样"的答案。如果答案
  是"全完了"，我们重新设计。
- 每一次迁移都是一次性向前。我们**不**发"version flag"保留旧行为
  ——新形状一落地，旧的就删除。

---

## 反原则（我们明确**不**是什么）

- **不是 OTA**。我们不抽预订佣金。商家收他们收的，赚他们赚的，
  减去可选的托管费。
- **不是钱包**。我们从不托管用户私钥、商家私钥、USDC。
- **不是库存管理 SaaS**。merchant-agent 模板自带默认库存模块，
  但库存是**每个商家自己的问题**，不是我们的数据库。
- **不是封闭生态**。任何 agent 客户端（Claude、ChatGPT、Cursor、
  自定义）都能 install 我们的 SKILL.md，**不需要向我们要 API key**。

---

## 怎么用这份文档

当你即将做一个非平凡决定（新 endpoint、新合约字段、新依赖、对现有
规则的新例外），重读原则。如果你的改动与某条原则冲突，**在 PR
描述里以这种形式明确论证例外**：

> "违反原则 N 因为 <原因>。在此可接受因为 <论证>。
> 缓解措施：<这个例外不会复利到下一个决定>。"

大部分违反应该过不了这个自检，最终被替换成不同的设计。少数能通过的，
作为已知例外记录到 `08_OPEN_QUESTIONS.zh.md` 里。
