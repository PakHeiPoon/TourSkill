# 商业模式 —— 协议免费，托管付费

> 引用：[00_PRINCIPLES.zh.md](./00_PRINCIPLES.zh.md)、[01_TARGET_ARCHITECTURE.zh.md](./01_TARGET_ARCHITECTURE.zh.md)。

这份文档定义 TourSkill 怎么挣钱。商业模式形态对架构是**承重的**：
它决定平台托管 runtime 要做什么、自托管用户免费拿到什么、我们永远
不会收钱的边界在哪。

---

## 1. 北极星

**协议免费。便利付费。**

我们明确**不**是 OTA。**永远不抽预订佣金**。**永远不托管用户资金**。
我们永远不把自己塞进 user-agent 与 merchant-agent 之间任何有经济
权重的流程里。

我们卖的：**跑一个 merchant-agent 的运维痛苦**。托管、升级、监控、
支持、扩缩容、私钥管理 UX、多门店汇总、自定义 skill 开发。这是 SaaS
定价，按"每商家每月"计费，**完全独立于他们的预订流水**。

这把我们放在好同伴里：

| 开放标准 | 免费 | 付费托管产品 |
|---|---|---|
| HTTP / SMTP | RFC 文档 | Cloudflare、Mailgun、Postmark |
| Postgres | postgresql.org | Supabase、Neon、Crunchy |
| Next.js / React | 框架本身 | Vercel、Netlify |
| Bitcoin / Ethereum | 链本身 | Coinbase、Alchemy、Infura |
| **A2A / ERC-8004 / x402** | 协议本身 | **TourSkill（我们）** |

这个模式经久不衰：开放标准赢采用率，便利产品赢收入。**我们故意站
那一边**。

## 2. 两条赛道（重申原则 6）

| | 自托管 | 平台托管 |
|---|---|---|
| 谁跑 merchant-agent 进程 | 商家 | TourSkill |
| 托管成本 | 商家付云账单（~$5-50/月 Vercel/Render/Fly）| 包在订阅里 |
| 钱包私钥托管 | 商家 | 商家*（我们从不托管）* |
| 库存数据位置 | 商家自己的 DB | TourSkill 多租户 Postgres，按 tenant 隔离 |
| 升级 / 安全补丁 | 商家自己跑 `git pull` | 我们自动推 |
| 外部 API 接口 | 完全相同 | 完全相同 |
| ERC-8004 注册 | 同一流程 | 同一流程 |
| 商家付给 TourSkill | $0 | 订阅（见下） |

**这条架构规则对两边都是杀手锏**：商家可以从平台托管开始，长大后
转自托管，**他们的客户（user-agent）完全察觉不到**。因为 agent-card
URL 只是一个域名，DNS 让它可携带。

这是合约：我们**只在我们更好的时候**赚他们的订阅，**不靠 lock-in**。

## 3. 定价档位（提议；待市场验证）

价格全部以 **USDC 等价**计，所以可链上结算（如有需要）。商家也能
通过标准渠道用法币付，**头部价格相同**。

### Tier 0 —— Solo（免费）
- 1 个商家
- 1 个门店
- 每月 ≤ 100 笔 settled booking
- 默认 skill 集（无自定义 skill）
- TourSkill 子域名（`<slug>.merchants.tourskill.paking.xyz`）
- 仅社区支持

**为什么免费**：独立宿主、单门店 B&B、"参加黑客松"用例。**网络效应的发现表面**。我们承担成本；网络效应是回报。

### Tier 1 —— Studio（$29 USDC/月）
- 最多 3 个门店
- 每月 ≤ 1,000 笔 settled booking
- 通过无代码表单做自定义 skill（日历覆盖、特殊膳食标签、自定义取消政策窗口）
- 自定义域名支持
- 邮件支持，48 小时 SLA

**目标客户**：小型 B&B 连锁、单餐厅集团、小众旅游运营商。

### Tier 2 —— Studio Plus（$99 USDC/月）
- 最多 10 个门店
- 每月 ≤ 10,000 笔 settled booking
- 通过开发者 hook 做自定义 skill（TypeScript 回调函数，沙箱化）
- 多门店汇总仪表盘（"显示我 8 家酒店的入住率"）
- Webhook 集成（与商家的 PMS 同步 —— Opera、Mews、Cloudbeds 等）
- 优先支持，24 小时 SLA

**目标客户**：区域酒店集团、连锁餐厅、多公园景区运营商。

### Tier 3 —— Enterprise（定制，起步 $499/月）
- 无限门店 / 预订
- 独立基础设施（单租户部署）
- 与商家现有系统的定制集成
- SOC 2 / ISO 27001 认证证明
- 专属 CSM、4 小时 SLA、工作时间电话支持
- 服务积分保证

**目标客户**：品牌连锁（万豪加盟集团、景区连锁）、运营子注册表的城市旅游局、想从老旧基础设施迁出的大型 OTA。

## 4. 永远免费的（即使在 Tier 0 上）

这是硬合约 —— 这些东西**无论哪个档位都不收费**：

- **协议本身**。ERC-8004 合约公开；agent-card.json spec 开放；merchant-agent 模板 MIT 协议；SKILL.md 公开服务。
- **自托管**。商家随时可以跳出我们的托管自己跑。**永远不会**用 feature flag 或"premium-only 协议扩展"卡这一条。
- **通过 TourSkill registry indexer 发现**。自托管和平台托管商家在 `/v1/discover` 结果中**一视同仁**。
- **用户侧 agent 安装**。终端用户（和他们的 user-agent）**永远不付 TourSkill 一分钱**。他们通过 x402 直接付给商家。
- **从我们平台迁出**。Tier 2 商家想离开，**一键导出**库存 + agent-card。我们帮他完成 DNS 切换。

**这是架构上的强制，不只是好心承诺** —— ERC-8004 registry 是我们不
拥有的公链状态；agent-card 由商家自己跑的代码服务；user-agent 在
没有我们参与下被用户安装。"协议免费"承诺**是系统的属性**，不是
营销话术。

## 5. 未来收入线（roadmap，不是 v1）

延伸同一逻辑（"为运维痛苦付费，从不为协议访问付费"）：

### 5.1 验证商家证明
我们作为（潜在多个）attestor 之一，签名"我们已验证这家商家在这个
地址确实存在"。按每次证明计费。**对商家是可选的**。在链上的
`ValidationRegistry` 里。

### 5.2 信誉 oracle
跨多源索引信誉（链上 settled booking + 可选的链外信号），向高量
agent 运营商（例如，每天 10K 次 discover-and-rank 查询的旅行聚合
agent）暴露付费 API tier。

### 5.3 自定义 skill 市场
第三方开发者发布可复用 skill 模块（"Stripe 退款处理器"、"多语言
礼宾"、"忠诚度积分发放器"）。我们按每模块上架收平价上架费。Tier 2+
商家可以混搭。

### 5.4 预订数据分析（保护隐私）
聚合、匿名化的预订数据给旅游局 / 市场研究者。商家自愿加入。**与加入
商家分成**。

### 5.5 0G 集成打包
把 0G Compute credits 和 0G Storage 配额打包进 Tier 1+ —— 让商家在
他们身份所在的同一链生态系统里跑 LLM 推理。在 0G 批发价上加 margin。

## 6. 我们**永远不会做的事**

这些是不可谈判的架构承诺，**不只是当下的策略**：

- **抽商家预订佣金**。无论 0.1% 还是 25%，都不抽。**只订阅**。
- **托管商家或用户资金**。所有 USDC 结算路径：escrow → 商家自己的钱包。**句号**。
- **把自托管商家从发现里藏起来**。两条赛道在 `/v1/discover` 里**永远等同曝光**。
- **把数据锁在付费 tier 后面**。即便 Tier 0 商家也拥有自己的库存；我们**不会**把它当作防止流失的人质。
- **未经同意卖商家数据**。匿名聚合需要 opt-in 加分成。

如果未来某个董事 / 投资人 / 联创推这五条任何一条，**答案是不**，
而且答案就在这一页上。

## 7. 经济上的开放问题

这些标记在 [08_OPEN_QUESTIONS.zh.md](./08_OPEN_QUESTIONS.zh.md) 里以后再决议。完整列在这里。

- **网络效应 vs 订阅 张力**。网络上商家越多，每个商家的 listing 越
  有价值 —— 但只有 Tier 0 免费，限制采用率。是不是该有"赚到钱再
  付费"变体，超过某个预订量阈值才从 Tier 0 → Tier 1？
- **支付通道 facilitator 费**。x402 facilitator 节点可以为验证支付
  证明收一点小费。如果我们自己跑 facilitator，每次调用有微小收入
  线。值不值得，是噪音吗？
- **token / 忠诚度机制**。很多 web3 marketplace 加 token。**v1 拒绝**
  —— token 引入监管复杂度，把激励对齐到投机而不是服务质量。**只在
  PMF 要求时才重新考虑**。

---

## 8. 北极星，再说一遍

> "开放协议，付费托管"。我们 ship 的每个特性都要对照这个标准来
> 评判：它属于*付费托管*的活儿，还是其实是*协议层级*的工作（应该
> 免费）。**有疑虑时，往免费一边偏**。
