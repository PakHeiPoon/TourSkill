# 待拍板的开放问题

> 引用：`docs/architecture/` 里所有其他文档。

这是**还没决定 / 我自己拿不下来 / 需要用户输入**的问题清单。每条带
问题、我看到的取舍、当前默认答案、什么时候要重新审视。

按需要回答的顺序排（最早的阻塞在前）。

---

## Q1. 索引器：进程内 还是 外部服务？

**问题**。新的 ERC-8004 索引器（回放 AgentRegistered / AgentUpdated
事件并填 `agents` 表）—— 在现有 FastAPI 进程里跑后台任务，还是单独
跑一个长期服务？

**取舍**。
- **进程内**（FastAPI 里的 asyncio 后台任务）：一次部署、运维更简
  单、白嫖 Vercel Fluid Compute。**风险**：Vercel 杀闲置函数；每次
  冷启动都要从最后区块回放。在繁忙链上**很浪费**。
- **外部**：跑在 Fly.io / Railway / 一台小 VPS 上的 Node 小服务，
  与 Base RPC 持久连接，写 Supabase。**部件多，但是索引器该有的
  形状**。

**我的默认**：外部 Node 服务（~50 LOC + Drizzle），$7/月 Fly.io。
**索引器是长期 watcher；serverless 函数不是**。

**何时重审**：如果外部索引器在规模上变成"每商家成本"问题（不该会
发生，全局单索引器服务所有商家），或者我们把链主切到不支持 log
filter 的链。

## Q2. 单全局索引器 vs 去中心化索引器？

**问题**。该有**恰好一个** TourSkill 跑的索引器，还是**鼓励 / 要求
多个独立索引器**做韧性？

**取舍**。
- **一个全局**：更简单、成本更低、更易调试。**发现的单点故障**；
  没我们链照样转。
- **多个**：任何人都能跑索引器，客户端选择查谁（或查多个并合并）。
  更鲁棒，更复杂。

**我的默认**：v1 一个全局索引器。**记录 schema + 索引器代码**让任何
人都能跑自己的（Node 服务开源）。**v2 真有需求时**（例如企业客户
要私有索引器）再向多索引器移动。

**何时重审**：TourSkill 的索引器 down 超过 1 天，或生态合作伙伴要
data feed。

## Q3. agent-card 哈希值得存吗，还是只存 URI？

**问题**。[02_ERC8004_CONTRACT_DESIGN.zh.md](./02_ERC8004_CONTRACT_DESIGN.zh.md) §2.1 在 IdentityRegistry 里同时存
`agentCardURI` 和 `agentCardHash`。哈希值得这点存储成本吗？

**取舍**。
- **带哈希**：客户端**不会被毒化的 URI 骗**（商家 CDN 服务被改的
  内容）。每个 agent 多 1 个 slot（Base 上 ~$0.001）。
- **不带哈希**：省 slot，**信任商家服务一致 JSON**。**他们 CDN 被
  攻陷 = 每个消费者都中招**。

**我的默认**：**保留哈希**。成本微不足道；安全收益是真的（想想 DNS
劫持、过期域名接管、恶意 CDN）。

**何时重审**：永不。**这一条已 settled —— 哈希保留**。

## Q4. 取消政策：也上链？

**问题**。取消政策当前在 agent-card.json（链下）。**也**编码到链上
让 escrow 合约能程序化强制部分退款？

**取舍**。
- **链上**：合约强制的退款 tier；不需要信任 merchant-agent 自己履行
  自己政策；更确定性。
- **仅链下**：escrow 合约保持简单；merchant-agent 自己跑退款逻辑并
  调 `escrow.refund()`；商家技术上能违反自己政策但**信誉会跳水**。

**我的默认**：**v1 仅链下**。理由：(a) escrow 合约保持最小可审计、
(b) 政策规则复杂（分级时间、自由取消窗口、带费用的部分退款）链上
编码加很多状态、(c) 商家违反自己公布政策的会被信誉系统 + 未来链上
争议捕获。

**何时重审**：看到信誉系统抓不到的违反政策模式，或大客户（城市旅游
局、大连锁）合规要求。

## Q5. 多门店支持 —— 单 agent 还是 一门店一 agent？

**问题**。一个 50 家门店的酒店连锁 —— 跑**一个** merchant-agent 暴露
50 套库存（多门店），还是**50 个**单独的 merchant-agent（链上各自
注册）？

**取舍**。
- **一 agent 多门店**：连锁运维简单，一份 agent-card 带 `properties[]`
  扩展；链上 footprint = 1 个 agent 费。但概念混乱：哪个门店是
  agent？不同城市 → 不同信誉。
- **一门店一 agent**：概念更清晰，每门店有自己的信誉、自己的
  agent-card；链支付 50× 注册。汇总仪表盘由平台侧的托管 tier 处理。

**我的默认**：**一门店一 agent**。强制清晰分离，匹配用户直觉
（"我评的是这家东京万豪，不是品牌"）。**连锁付 N×$0.005 注册 gas，
小钱**。

**何时重审**：Tier 3 企业客户强烈反对。也许通过 ValidationRegistry
加"品牌证明"，连锁证明"这 50 个 agent 都是我"。

## Q6. 幂等窗口 —— 多长？

**问题**。`create_booking` 带 `Idempotency-Key` 调用时，merchant-
agent 记结果多久、重试时返缓存？

**取舍**。
- **长窗口（24h+）**：对所有重试模式安全；吃 DB 空间。
- **短窗口（5m）**：footprint 小；超过窗口的 user-agent 重试可能
  双订。

**我的默认**：**24 小时**。存储成本可忽略。真实 flaky 网络上的重试
有时几小时后才发生。

**何时重审**：负载测试发现幂等表膨胀。

## Q7. Skill：多少个、哪些必须？

**问题**。每个 hotel 类型商家是不是必须暴露所有标准 skill
（`check_availability`、`get_rates`、`create_booking`、
`get_cancellation_policy`）？如果商家只想做发现-only 不支持预订呢？

**取舍**。
- **强制全集**：user-agent 一致 UX，无意外（"这商家说自己是 hotel
  但你订不了"）。
- **可选**：更灵活；有些商家只想要目录式露出；agent-card 声明能干
  啥。

**我的默认**：**可选**。agent-card 列 agent 支持的 skill；user-
agent 的 discover 查询能按必须 skill 过滤
（`/v1/discover?skills=create_booking`）。**与 spec 其他部分一致**。

**何时重审**：UX 测试显示用户在同一列表里被"浏览" vs "预订"商家
搞糊涂。也许加个可发现性徽章（"可预订"）。

## Q8. 争议仲裁 —— v1 谁来？

**问题**。[05_X402_PAYMENT_FLOW.zh.md](./05_X402_PAYMENT_FLOW.zh.md) §3 说 v1 我们（TourSkill）是仲裁者。
**怎么搭这个？**

**取舍**。
- **单个 admin EOA**：最简单。一个人决定。**风险**：那人可能被社工
  或胁迫。
- **多签（2-of-3 或 3-of-5）**：团队内分散，抗攻击。决定更慢。
- **从 day 1 用外部仲裁服务**（Kleros / Reality.eth）：最难搭；最
  正当。

**我的默认**：**2-of-3 多签，联创做签名者**。**目标争议解决时间
SLA 7 天**。明确 roadmap 在 v2 迁到 Kleros / 类似。

**何时重审**：解决第 10 个争议时；**模式数据**告诉我们什么是难的。

## Q9. 0G 集成深度？

**问题**。我们在 AgentDemo 用 0G Compute 作为推理 provider（可选）。
是不是也用 0G Storage 做 agent-card 托管？做预订 tx 归档？做信誉反馈
存储？

**取舍**。
- **重 0G 集成**：强化"0G 生态"叙事；**对 0G 寿命的技术下注**。
- **轻集成**：耦合更少；如果 0G 的产品/团队轨迹变了，我们没事。

**我的默认**：**轻**。0G Compute **保持作为可选推理 provider**
（商家可以选任何 OpenAI 兼容的）。0G Storage **roadmap 上做 Tier 2+
特性的"永久信誉归档"**，不是 v1。其他接触点保持 vendor 无关。

**何时重审**：0G Storage 成熟带稳定客户端库 + 我们在反馈归档上撞到
Supabase 定价痛苦。

## Q10. 多租户 agent runtime —— 隔离多强？

**问题**。TourSkill 平台托管 merchant-agent 时，租户隔离多强？同
Hono 进程加 Postgres 行级过滤？每租户独立进程？每租户独立 VM？

**取舍**。
- **进程级隔离**：扩展最简单、成本最低。**一个租户的 bug 能搞挂
  整个进程**。
- **每租户独立进程**：隔离更强，**成本随租户数量线性扩展**。Vercel
  上不便宜。
- **每租户独立 VM**：隔离最强，**非常贵**，仅对 Tier 3 企业客户有
  道理。

**我的默认**：**Tier 0 + Tier 1 + Tier 2 共享一个进程池 + Postgres
行级过滤**。**Tier 3 拿独立 VM**。同一份代码两种模式跑；区别在
orchestrator。

**何时重审**：租户 bug 或 RCE 事故；首次审计；或第一个要 SOC 2 证据
的 Tier 3 客户。

## Q11. 自定义 skill —— 沙箱化 还是 审核？

**问题**。Tier 2+ 特性：商家能写自定义 skill 代码（TypeScript 回调）。
**沙箱**跑这代码（Cloudflare Workers / Deno permission gate）还是
**审核** / 审计每个商家的代码？

**取舍**。
- **沙箱**：扩展到 N 个商家不需要每商家审核工作。漏掉沙箱预料不到的
  攻击。
- **审核**：每个商家提交都过代码 review。**不扩展；偏向不能负担等待
  的小商家**。
- **两者**：先沙箱，更高 tier 给"可信、审过"自定义代码。

**我的默认**：v1 没有自定义代码特性；无代码表单对 Tier 1/2 够了。
自定义代码**仅 Tier 3** 且部署前**人工审核**。

**何时重审**：Tier 2 流失数据显示商家因为不能自定义而离开；或
Cloudflare Workers permission 模型改进。

## Q12. "如果 Coinbase 弃了 x402 怎么办？"

**问题**。x402 当前是 Coinbase 主导的标准。如果 Coinbase 失去兴趣
或转向，我们的连续性方案是什么？

**取舍**。
- **押注 x402**：现在最简单；和 crypto 最大支付玩家对齐。
- **建通用支付方法抽象**：更多代码，未来证明但**过早**。

**我的默认**：**押 x402**，但**支付集成写在干净接口（merchant-agent
模板里的 `PaymentMethod` trait）后面**，让未来"x402 替代"是单模块
替换。接口镜像 402 wire format；如果 HTTP 402 本身将来标准化成不同
形状，我们适配 wire format 保持接口。

**何时重审**：Coinbase 公开退出 x402，或某个等价 EIP 出现并获得
牵引力。

## Q13. v2 货币：USDT、EURC、原生 ETH？

**问题**。按 [05_X402_PAYMENT_FLOW.zh.md](./05_X402_PAYMENT_FLOW.zh.md) §6，v1 仅 Base 上的 USDC。
**v2 优先级是什么？**

**取舍**。
- **USDT 优先**：全球最常用稳定币，**尤其在亚洲**（我们的初期市场）。
- **EURC 优先**：补充 USDC、Coinbase 原生、**欧盟监管更清晰**但市场
  小。
- **原生 ETH**：集成更简单，匹配更广 crypto 生态的"按调用付费"模式；
  但**波动对预订报价不好**（"今天 3640 ETH，明天 3580 ETH"）。

**我的默认**：**USDT 接下来**。亚洲市场重要；我们在澳门 / 香港 /
内地。

**何时重审**：v1 ship 后我们有真实预订数据显示摩擦在哪。

## Q14. 地理覆盖：仅链上 还是 法币 ramp 的 KYC？

**问题**。最终我们想要法币入金通道（Visa → USDC → escrow）触达非
crypto 原生客户。**这把我们拉进 KYC / AML 领域，破坏"TourSkill 永
不托管"吗？**

**取舍**。
- **仅 crypto**：永不托管，永不需要 KYC。**TAM 限于 crypto 流利
  用户**。
- **与 on-ramp 提供商合作**（Coinbase Onramp、MoonPay）：**他们做
  KYC**；用户带着 USDC 落地，**我们不碰**。**TAM 显著扩大**。
- **建自己的 ramp**：永不。别。

**我的默认**：**合作伙伴 on-ramp（Coinbase Onramp）**等其他栈跑通
后。我们不托管。**合作伙伴是 ramp**；我们只在 user-agent UX 里
嵌入他们的 widget。

**何时重审**：v1 ship 后 PMF 表明 TAM 扩大是瓶颈。

## Q15. 无聊的：用 Foundry 还是 Hardhat？

**问题**。智能合约框架。

**取舍**。Foundry 更快、原生 Solidity、现代工具链。Hardhat 对许多
开发者更熟悉、JS 工具集成更多。

**我的默认**：**Foundry**。**测试更快 = 测试更好**。工具链精简。

**何时重审**：永不。**Foundry 是 2026 年的对的选择**。

## 怎么用这份清单

每个问题有默认。**默认是 "够好可以开建" 的程度**。我们在执行中撞到
每个时：

- 同意默认 → **不需要决定**；代码反映它。
- 默认要变 → **更新本文档**带新决定 + 推理，标注其他要跟进的文档。
- 建造中冒出的新问题 → **进同一文件，同结构**。

**这是活文件，不是一次性 checklist**。**更新版本控制；季度或主要
决定落地时审一次**。
