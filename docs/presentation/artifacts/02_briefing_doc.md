# Concourse：去中心化智能体（A2A）旅游商业协议简报

## 执行摘要（Executive Summary）

Concourse 是一个受比特币白皮书启发、专为智能旅游智能体（Intelligent Agents）设计的去中心化协议。其核心是一个去中心化的“黄页”注册表，允许商家直接发布可执行的“技能”（Executable Skills），使消费端智能体能够绕过中心化在线旅游平台（OTA）的租金提取，直接与商家进行点对点（P2P）交互。

本项目在计算机科学领域，特别是智能体研究维度上，涵盖了感知、知识表示（SKILL.md）、决策、行动、多智能体协调以及信任验证等关键环节。其技术架构采用了链上身份锚点与离线可变配置相结合的三层模式，并通过创新的“一次签名，永久治理”模式解决了智能体安全持钥的难题。

---

## 1. 核心动机：为何旅游业是智能体的最佳实验场？

当前旅游市场被少数中心化 OTA 深度垄断。商家标价 800 元的房间，用户需支付 1000 元，中间 100 元差价被平台作为中介费提取。智能体如果不打破这种 API 授权的封闭生态，将沦为平台租金提取的帮凶。

**旅游业的科研价值：**
*   **多模态服务束：** 一次旅行涉及餐饮、酒店、交通等多种交互模式（查询/预订/支付/确认）。
*   **丰富的自然语言需求：** 用户意图通常是模糊的（如“杭州两日游”），对智能体的“理解→规划→委托”链路提出了极高要求。
*   **去中心化需求迫切：** OTA 的寻租行为透明且可见，是验证 A2A 直接交易逻辑的直观场景。

## 2. 系统定义与三层架构

Concourse 将旅游商业从“依赖平台”推向“智能体自由”。

### 三层技术架构
为了平衡区块链的不可篡改性与旅游业信息的时效性，系统分为三层：

| 层次 | 组件 | 职能 |
| :--- | :--- | :--- |
| **身份层** | 链上锚点 (On-chain) | 保持身份的可验证性与不可篡改（使用 0G 网络）。 |
| **配置层** | 离线配置 (Off-chain) | 保持商家配置（如营业时间、菜单）的可变性与低成本修改。 |
| **协议层** | SKILL.md | 将智能体接口标准化，使其具备可移植性。 |

---

## 3. SKILL.md 协议：面向 LLM 的合同

Concourse 弃用了传统的 OpenAPI/JSON 格式，改用 **Markdown (SKILL.md)** 作为智能体面对面的合同。

*   **设计逻辑：** LLM 原生具备阅读 Markdown 的能力。SKILL.md 旨在通过“上下文学习”（In-context learning）让智能体理解动词、HTTP 契约、认证规则及双向提示词。
*   **安装便捷性：** 用户只需向智能体输入安装链接（如 `https://api.tourskill.paking.xyz/skills/user-client/SKILL.md`），智能体即可加载合约并开始调用注册表。
*   **关键功能：** 规范智能体禁止违反的硬性规则（如“未经用户确认严禁代签”）。

---

## 4. “一次签名”启动仪式：解决密钥持有难题

**核心矛盾：** 商家智能体不能持有钱包私钥（否则私钥泄露会导致资金损失），但又必须代表商家修改配置（如暂停营业、更新菜单）。

**Concourse 的解决方案：**
系统采用**草稿-URL 移交模式**（Draft-URL handoff pattern），将“持钥环境”（浏览器+MetaMask）与“行动环境”（智能体+Bearer Token）彻底分离：

1.  **协作拟稿：** 智能体与商家主理人沟通，生成配置草稿。
2.  **移交签名：** 智能体生成一个一次性签名链接，主理人在浏览器中打开。
3.  **双重签名仪式：**
    *   **链上交易：** 调用 `MerchantRegistry.register` 锚定身份（产生 Gas）。
    *   **链下挑战：** 进行 EIP-191 `personal_sign` 签名，换取一个有效期 30 天的 Bearer Token。
4.  **持有令牌：** 智能体随后仅持有 Token 进行 API 调用，主理人无需反复操作钱包。

## 5. 安全深度调查：从漏洞到 EIP-191 修复（技术巅峰）

本项目的核心安全贡献在于对智能体身份验证机制的迭代。

### 5.1 漏洞分析：错误的“公开标识符即秘密”
在开发早期，项目曾使用 `X-Wallet-Address: 0x...` 作为身份验证头。
*   **缺陷：** 钱包地址在区块链浏览器上是公开数据。
*   **攻击复现：** 攻击者只需通过简单的 `curl` 命令，伪造他人的钱包地址头，即可恶意篡改他人的商家信息（如修改邮箱、强行关闭店铺）。

### 5.2 修复方案：EIP-191 挑战-响应机制
通过引入标准的挑战-响应逻辑，成功防御了伪造攻击。

**攻击者击败矩阵（Attacker Defeat Matrix）：**

| 攻击方式 | 修复前（旧版） | 修复后（Bearer Token） |
| :--- | :--- | :--- |
| 伪造 X-Wallet-Address 头部 | 200 OK，篡改成功 | 401 未授权 |
| 签署另一个钱包的 Nonce | 不适用 | 403 签名恢复失败 |
| 重放已失效的 Nonce | 不适用 | 400 无效或过期 |
| 跨商家使用合法 Token | 不适用 | 403 钱包不匹配 |

**结论：** 本案例证明了在智能体系统中，**公开标识符绝不能作为授权凭证**。

## 7. 路线图与未来研究方向

### 近期（Next）
*   **去中心化信誉系统：** 建立基于链上签名的评论机制，解决廉价刷单问题（Sybil-resistance）。
*   **x402 支付：** 引入 HTTP 402 原生微支付，使商家技能可以按次计费。

### 中期（Soon）
*   **MCP 桥接：** 将 Concourse 暴露为 MCP（Model Context Protocol）服务器，适配 Claude Desktop 等主流智能体工具。
*   **SIWE 升级：** 采用 EIP-4361 标准格式替换自定义挑战消息。

### 长期（Later）
*   **智能体谈判：** 研究两个 LLM 智能体在有限诚实条件下的价格博弈与博弈论平衡。
*   **可验证配置文件：** 每次 PATCH 后重新在链上锚定 Profile Hash，实现离线数据的完整性证明。
