# Concourse 学习指南：智能体对智能体（A2A）去中心化旅游协议

本指南旨在为“智能智能体（Intelligent Agents）”课程的演示提供深度学术支持。Concourse 是一个基于区块链和 AI 技术、受比特币白皮书启发的去中心化旅游商业协议，旨在通过 A2A（智能体对智能体）交互消除中介剥削。

---

## 一、 核心概念与理论基础

### 1. 什么是 A2A？为什么智能体需要开放注册表？
*   **A2A (Agent-to-Agent)：** 指代表用户的消费端智能体与代表商家的服务端智能体直接进行发现、协商、预订和支付的过程，无需经过中心化平台（如 OTA）。
*   **开放注册表的必要性：** 
    *   **打破垄断与寻租：** 现有的在线旅游平台（OTA）通过控制排名算法和 API 闸门提取巨额租金（如 800 元的房间显示为 1200 元再返券）。
    *   **发现与信任：** 如果没有一个去中心化且不可篡改的“黄页”，智能体将无法跨越平台边界找到合法的服务商，或者只能在封闭的生态系统（Walled Gardens）中运行。
    *   **互操作性：** 开放注册表为所有智能体提供了一致的身份验证和能力描述标准。

### 2. 三层架构设计
为了平衡区块链的不可篡改性与旅游业务的灵活性，Concourse 采用了三层分离架构：

| 架构层级 | 功能描述 | 技术实现 |
| :--- | :--- | :--- |
| **链上锚点层** | 确保身份永久、可验证且不可篡改。 | 0G Galileo 测试网上的智能合约 |
| **链下 Profile 层** | 存储易变数据（如菜单、营业时间、价格），保证更新低延迟、零成本。 | FastAPI + Supabase |
| **智能体合约层** | 定义智能体如何安装、调用和授权服务。 | SKILL.md (Markdown 协议) |

### 3. SKILL.md：可安装的智能体合同
SKILL.md 是 Concourse 的技术核心创新之一。它不是传统的 OpenAPI/JSON 规范，而是 LLM 原生的 Markdown 文档。
*   **为什么用 Markdown？** LLM 能够原生理解 Markdown 语义。它结合了函数调用的严谨性和人类可读的入职引导（Onboarding）。
*   **功能：** 指定智能体支持的动词（发现/调用/更新）、HTTP 合约、钱包绑定协议以及双语 Prompt 模板。

## 二、 安全架构与身份认证

### 1. “一次签名，永久治理” (Sign Once, Govern Forever)
智能体研究的一个核心挑战是：**智能体不应持有用户的私钥**。Concourse 的解决方案是：
*   **注册阶段：** 商家只需在浏览器中通过 MetaMask 签署两次：一次链上交易（身份锚定），一次 EIP-191 挑战-响应（生成 30 天有效期的 Bearer Token）。
*   **治理阶段：** 之后，智能体仅持有该 Token 即可代表商家执行更新菜单、暂停营业等操作，无需再次请求私钥签名。

### 2. 漏洞案例研究：X-Wallet-Address 的失败
*   **初期错误：** 项目早期使用 `X-Wallet-Address: 0x...` 作为认证 Header。
*   **安全威胁：** 钱包地址在区块链浏览器上是**公开数据**。攻击者只需获取受害者的地址，即可通过简单的 `curl` 请求篡改其商户信息。
*   **核心教训：** **公开标识符不是认证凭证（Public identifier is not a secret）。**

### 3. EIP-191 修复方案与攻击击败矩阵
通过引入服务器签发的挑战（Challenge）和客户端的加密响应，Concourse 成功防御了多种攻击：

| 攻击类型 | 修复前表现 | 修复后（挑战-响应）表现 |
| :--- | :--- | :--- |
| 伪造钱包地址 Header | 200 OK，篡改成功 | 401 Not authenticated |
| 使用他人钱包签名随机 Nonce | N/A | 403 Signature does not recover |
| 重放已失效的 Nonce | N/A | 400 Invalid or expired |
| 攻击者使用自己的合法 Token 尝试修改他人商户 | N/A | 403 Wallet does not own |

## 三、 未来路线图与研究议程

*   **信誉系统（Reputation）：** 每次智能体调用后留下链上评价，通过 Sybil-resist 机制（如支付收据证明）防止刷单。
*   **x402 支付：** 引入 HTTP 402 Payment Required 状态码，实现智能体间的微支付。
*   **MCP (Model Context Protocol)：** 兼容 Anthropic 提出的标准，让所有 MCP 智能体（如 Claude Desktop）能直接发现并调用 Concourse。
*   **SIWE (Sign-In With Ethereum)：** 采用 EIP-4361 标准，统一认证消息格式。

## 四、 简答练习题 (Short-Answer Quiz)

**Q1: 为什么 Concourse 选择在 0G Network 上部署？**
**A:** 0G Network 是模块化的 AI 原生链，提供去中心化存储和计算支持，适合未来实现链上推理证明和高频的商户状态更新。

**Q2: 描述 SKILL.md 如何实现“零配置”安装？**
**A:** 智能体只需读取一个公共 URL（如 api.tourskill.paking.xyz/skills/user-client/SKILL.md），解析其中的 Markdown 指令，即可学会如何查询注册表并执行特定类别的商户技能。

**Q3: 在 A2A 交互中，智能体如何进行“个性化排序”？**
**A:** 智能体不盲目信任注册表的默认顺序，而是根据其掌握的用户上下文（如过敏原、预算偏好、历史记录）对搜索结果进行二次加权排序。

## 六、 教授可能提出的 Q&A 预测及回答

**Q: 既然数据存储在 Supabase 这样的中心化数据库，这还算“去中心化”吗？**
**A:** 身份是去中心化的（链上锚点）。架构采用了“身份/状态分叉”设计。关键在于：身份由链上地址持有，且每次更新后的 Profile Hash 计划回填至链上。这种分层是为了在旅游行业的高频更新（如菜单变动）与区块链的成本/延迟之间取得平衡。

**Q: 如何防止商家注册虚假信息？**
**A:** 目前依靠链上身份的不可篡改性建立可回溯性。未来路线图包含“信誉系统”，只有产生过真实支付（通过 x402 协议）的智能体才能生成有效评价，从而在博弈论层面通过经济成本抑制虚假信息。

**Q: 如果智能体（如 LLM）产生幻觉，误订了不可取消的酒店，责任归谁？**
**A:** Concourse 的 SKILL.md 定义了严格的“硬规则”：智能体在涉及不可逆转的支付或违背用户历史偏好时，必须通过“草稿-审核”模式（Collaborative Draft）由用户进行最终确认。

## 七、 核心术语表 (Glossary)

*   **A2A:** Agent-to-Agent，智能体对智能体交互。
*   **SKILL.md:** 专为 LLM 设计的可安装技能说明文档。
*   **EIP-191:** 以太坊的一种签名格式，用于证明用户对特定消息的所有权而不触发交易。
*   **SIWE:** Sign-In With Ethereum，使用以太坊钱包登录的标准协议。
*   **MCP:** Model Context Protocol，旨在统一智能体与工具/数据源交互的协议。
*   **x402:** 基于 HTTP 402 状态码的智能体间自动支付协议。
*   **Profile Hash:** 商户资料的哈希值，用于验证链下数据是否被非法篡改。
